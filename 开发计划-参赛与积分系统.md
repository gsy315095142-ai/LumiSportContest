## 二、阶段 1：后端数据模型与赔率重构

### 2.1 目标

- 用户数据结构增加 `rating`（选手积分）
- 比赛状态增加 `redPlayer`、`bluePlayer`（参赛选手）
- 移除段位逻辑，实现基于积分的赔率计算

### 2.2 改动文件

| 文件 | 改动 |
|------|------|
| `server/index.js` | 见下文 |

### 2.3 具体改动

1. **用户数据**：`users.json` 中每个用户增加 `rating`（默认 1500），新用户或老用户缺失时补 1500
2. **gameState**：`createFreshGame()` 增加 `redPlayer: null`、`bluePlayer: null`
3. **赔率函数**：新增 `getOddsByRating(redRating, blueRating)`，替代 `getOdds(redTier, blueTier)`
   - 实现 5.3 赔率公式（积分差线性映射，1.1~2.0）
4. **buildGameInfo**：`gameInfo` 改为返回 `redRating`、`blueRating`、`odds`，移除 `redTier`、`blueTier`
5. **移除**：`TIERS`、`ODDS_MATRIX`、`getOdds` 等段位相关代码

### 2.4 依赖

- 无前置依赖

## 三、阶段 2：后端参赛 API 与结算逻辑

### 3.1 目标

- 参赛确认 / 撤销 API
- 下注时校验「已参赛则禁止」
- 参赛确认时校验「有下注则禁止」
- 结算时更新选手积分、写入历史战绩

### 3.2 新增 API

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/contest/confirm` | POST | 确认参赛（body: { name, side }） |
| `/api/contest/cancel` | POST | 撤销参赛（body: { name }） |
| `/api/contest/status` | GET | 获取当前参赛状态（含红蓝选手、当前用户是否已参赛） |
| `/api/match-history/:name` | GET | 获取用户历史战绩 |

### 3.3 新增 Socket 事件

| 事件 | 方向 | 说明 |
|------|------|------|
| `contest:confirm` | 客户端→服务端 | 确认参赛 |
| `contest:cancel` | 客户端→服务端 | 撤销参赛 |
| `contest:update` | 服务端→全体 | 参赛状态变更（红蓝选手、可选项） |

### 3.4 结算逻辑

- 在 `game:settle` 和 `api/admin/settle` 中：
  - 若有红蓝选手，则按比分差额更新 `redPlayer`、`bluePlayer` 的 `rating`
  - 平局时双方积分不变
  - 写入历史战绩（`matchHistory.json` 或 `users.json` 内 `matchHistory` 数组）

### 3.5 数据存储

- 历史战绩：`server/data/matchHistory.json` 或扩展 `users.json`
- 结构示例：`{ matchName, gameType, redPlayer, bluePlayer, redScore, blueScore, redRatingChange, blueRatingChange, result, time }`

### 3.6 依赖

- 依赖阶段 1

- 互斥校验（有下注则禁止确认；已参赛则禁止下注在竞猜页）

### 5.2 新增文件

| 文件 | 说明 |
|------|------|
| `src/mobile/JoinPage.jsx` | 我要参赛页面 |
| `src/mobile/JoinPage.css` | 参赛页样式 |

### 5.3 逻辑要点

- 通过 Socket 或 REST 获取 `gameInfo`、`contestStatus`（红蓝选手、当前用户是否已参赛）

- 确认前调用 API 校验是否有下注
- 确认 / 撤销后通过 Socket 刷新

## 九、阶段 8：Unity 联动与联调

### 9.1 目标

- Unity 端需能获取红蓝选手积分与赔率
- 当前 Admin API 的 `gameInfo` 已包含 `redRating`、`blueRating`、`odds`
- 需确认 Unity 端如何获取：轮询 `GET /api/game` 或通过 Socket 订阅 `game:update`

### 9.2 改动范围

- **LumiSportContest**：确保 `gameInfo` 含 `redRating`、`blueRating`、`odds`，Admin 接口无需额外改动
- **Unity Server**：需在 `D:\gitProject\LumiSports\Server` 中修改展示逻辑，由段位改为积分与赔率

### 9.3 建议

- 先完成阶段 1~7，再与 Unity 联调
- 联调时需同时启动 LumiSportContest 与 Unity，确认数据流正确

## 十一、风险与回退

| 风险 | 应对 |
|------|------|
| 老用户无 `rating` | 读取时缺省补 1500 |
| 历史战绩数据量大 | 分页或限制条数（如最近 50 条） |
| Unity 端改动较大 | 先保证竞猜后端与手机端闭环，Unity 可后续接入 |
