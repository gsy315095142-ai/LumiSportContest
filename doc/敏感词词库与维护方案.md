# 敏感词词库与维护方案（弹幕）

> 版本：v1.0  
> 更新时间：2026-03-30  
> 适用范围：`/api/danmaku/send` 敏感词拦截（命中即拒绝发送）

---

## 一、现状与目标

### 1.1 当前现状

- 服务端已迁移为读取 `server/data/danmakuBlockedWords.txt`（每行一个词）；
- 运行时在 `server/index.js` 的 `danmakuContainsBlocked(text)` 执行包含匹配；
- 当前词条可持续扩充，避免在代码中维护大数组。
- 2026-03-30 已基于开源词库批量扩充，现有词条约 2000+（含去重清洗）。

### 1.2 本文目标

- 给出可直接使用的开源敏感词来源（中文为主、英文补充）；
- 明确“下载 -> 合并 -> 去重 -> 上线”的维护流程；
- 保持当前规则不变：**命中任一敏感词即拒绝发送**。

## 三、词库接入建议（保持“命中即拒绝”）

## 3.1 词库形态建议

- 建议新增本地词库文件（例如）：`server/data/danmakuBlockedWords.txt`
- 格式：一行一个词；空行忽略；支持 `#` 注释行。

## 3.2 运行时规则（沿用当前逻辑）

- 中文词：`text.includes(word)`；
- 英文词：统一小写后 `includes`；
- 命中即返回 `400`：`内容包含不当词语`；
- 保留现有长度与频控规则（1~40 字、5 秒 1 条）。
- 词库更新后可调用热重载接口：`POST /api/danmaku/reload-words`（无需重启）。

## 3.3 合并与去重规则

1. 先合并“中文主词库 + 中文补充词库 + 英文补充词库”；
2. 统一 trim（去首尾空白）；
3. 删除空词、重复词；
4. 人工剔除明显误伤词（例如正常语义高频短词）；
5. 生成项目最终词库文件并纳入版本管理。

---

## 四、维护流程（建议）

1. 每周或每两周更新一次上游词库；
2. 更新词库文件后，调用 `POST /api/danmaku/reload-words` 应用新词条；
3. 在测试环境灰度，观察误杀率；
4. 若误伤上升，优先通过“白名单”或词条剔除修正；
5. 每次更新记录版本号与来源快照（日期 + URL）。

---

## 五、风险与注意事项

- 开源词库通常“宁可多拦截”，可能出现误伤；
- 不同地区/场景对敏感词定义差异较大，应结合业务调整；
- 使用第三方词库时，需保留原仓库许可声明（如 MIT）。
