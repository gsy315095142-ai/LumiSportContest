# 选手积分与竞猜规则（2026-03-30）

> 适用范围：竞技玩法（如魔法拳王、魔法冰球等）的赛前赔率计算与赛后积分结算。

---

## 一、规则目标

- 不同竞技玩法独立维护积分，避免跨玩法互相干扰。
- 根据双方当前积分动态计算预期胜率与赔率。
- 赛后按胜负更新积分，体现“以弱胜强加分更多、输给强者少扣分”。

## 三、变量定义

| 符号 | 含义 | 默认值/说明 |
|------|------|-------------|
| `a` | 当前玩法下，我方赛前积分 | 初始值 `100` |
| `b` | 当前玩法下，对手赛前积分 | 初始值 `100` |
| `w` | 我方预期胜率 | 由积分计算 |
| `r` | 庄家服务费率 | `10%` |
| `p` | 积分浮动率（得分率） | `5%` |
| `B` | 我方获胜后的新积分 | 结算结果 |
| `C` | 我方失败后的新积分 | 结算结果 |

## 四、核心规则

### 4.1 玩法独立积分

- 同一选手在不同玩法中，积分互相独立。
- 示例：某选手在“魔法拳王”和“魔法冰球”应分别维护两套积分。

### 4.2 初始积分

- 每个玩法下，选手初始积分统一为 `100`。

### 4.3 预期胜率

`w = a / (a + b)`

说明：`w` 取值范围在 `(0, 1)`，用于赔率计算。

### 4.4 赔率计算

`我的赔率 = (1 / w) * (1 - r)`

其中 `r = 10%`。  
说明：服务费在赔率侧扣减。

### 4.5 胜负后积分结算

- 我方获胜：

`B = a * (1 + p * 对手胜率 / 自己胜率)`

- 我方失败：

`C = a / (1 + p * 自己胜率 / 对手胜率)`

其中 `p = 5%`。  
说明：该失败公式满足“对手越强，输掉后扣分越少；对手越弱，输掉后扣分越多”。

### 4.6 积分下限

- 任意玩法下，任意选手积分最低为 `1`。
- 每次结算后需执行下限钳制：`新积分 = max(1, 结算值)`。

### 4.7 胜负竞猜准入限制

- 当任意一方的胜利预期 `>= 90%` 时，该场次**禁止进行胜负相关竞猜**。
- 上述限制仅作用于胜负竞猜，不影响趣味竞猜（趣味竞猜仍可正常进行）。
- 手机端需提供明确的状态表现与提示文案（示例：“双方实力差距过大，本场暂不开放胜负竞猜，可参与趣味竞猜”）。

## 五、结算流程（单场）

1. 读取当前玩法下双方赛前积分 `a` 与 `b`。  
2. 计算预期胜率 `w`。  
3. 按服务费率 `r` 计算我方赔率。  
4. 比赛结束后按胜负结果，使用 `B` 或 `C` 更新我方积分。  
5. 将新积分写回该玩法积分池（仅影响当前玩法）。

---

## 六、快速示例

设：`a = 100`，`b = 150`，`r = 10%`，`p = 5%`。

- 预期胜率：`w = 100 / (100 + 150) = 0.4`
- 赔率：`(1 / 0.4) * 0.9 = 2.25`
- 若我方获胜：`B = 100 * (1 + 0.05 * 150 / 100) = 107.5`
- 若我方失败：`C = 100 / (1 + 0.05 * 100 / 150) ≈ 96.77`

结论：面对高于自己的积分对手，失败扣分相对温和。

## 七、实现与接入说明

### 7.1 服务端实现注意事项

- 计算时建议使用浮点并在展示层统一保留小数位（如 2 位）。
- 分母必须防止为 `0`（理论上积分不应为 `0`，但代码仍需兜底）。
- 所有读写必须按“玩法维度 + 选手维度”进行键控，避免串数据。
- 竞猜入口渲染时需先计算双方预期胜率；若任一方 `>= 90%`，胜负竞猜入口置灰/禁用，并展示限制说明。
- 手机端交互应区分“胜负竞猜不可用”与“趣味竞猜可用”的状态，避免用户误解为整场不可竞猜。

### 7.2 Unity 端后续接入改造说明

> Unity 端项目位于 `D:\gitProject\LumiSports\Server`。本次后端已完成规则升级，Unity 端需要按以下要点完成适配。

#### 7.2.1 数据结构改造（核心）

- 旧结构（历史方案）：选手单一积分 `rating`（全玩法共用）。
- 新结构（当前方案）：选手积分改为按玩法独立：

```json
{
  "ratings": {
    "hockey": 100,
    "boxing": 100
  }
}
```

- 如 Unity 端有本地缓存/镜像模型（DTO/VO），需将 `rating` 升级为 `ratings[gameType]`。
- `rating` 字段在服务端仅做兼容保留，后续逻辑应以 `ratings` 为准。

#### 7.2.2 需要消费的新字段（比赛状态）

Unity 若有展示竞猜信息（或推送给终端），需从 `/api/game` 或 `game:update` 中使用以下字段：

- `gameType`：当前玩法（`hockey` / `boxing`）
- `redRating`、`blueRating`：当前玩法下红蓝双方积分
- `redWinRate`、`blueWinRate`：双方预期胜率（0~1）
- `winBetEnabled`：是否允许胜负竞猜
- `winBetDisabledReason`：不可下注时的提示文案

#### 7.2.3 胜负竞猜限制联动

- 当 `winBetEnabled = false`（任意一方胜率 `>= 90%`）时：
  - 禁止发起/展示胜负竞猜入口；
  - 保留趣味竞猜入口；
  - UI 提示使用 `winBetDisabledReason`。

#### 7.2.4 Admin 结算接口调用说明

- Unity 继续调用既有流程即可：`/api/admin/configure` -> `/api/admin/lock` -> `/api/admin/settle`。
- 本次升级后，积分结算由服务端按 `gameType` 自动落到对应玩法积分池，无需 Unity 额外传入积分。
- Unity 只需保证每局 `gameType` 正确传入（或保持与当前局一致）。

#### 7.2.5 历史与排行榜适配

- 选手积分榜接口支持玩法维度查询：`/api/rating-rankings?gameType=hockey|boxing`。
- 若 Unity 展示战绩变更，可读取历史中的 `ratingGameType`、`redRatingBefore/After`、`blueRatingBefore/After` 字段，明确是哪种玩法下的积分变化。
