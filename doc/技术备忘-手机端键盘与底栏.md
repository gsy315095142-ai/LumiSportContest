# 技术备忘：手机端键盘与固定底栏（`/mobile`）

> 最后更新：2026-03-30  
> 适用：`src/mobile/MobileApp.jsx` 中带底部三按钮导航的壳层

---

## 一、现象

在手机端 `/mobile` 路由下，用户聚焦 **竞猜金额输入框**、**弹幕输入** 等控件时，底部固定导航（「我要竞猜 | 发弹幕 | 我要参赛」）会被 **错误地抬到键盘上方**，视觉上像整条底栏「跟着键盘浮起来」，而不是稳定贴在屏幕物理底边。

---

## 二、根因（为何仅靠 viewport / CSS 高度往往无效）

1. **`position: fixed; bottom: 0` 的参照系是「布局视口」**  
   键盘弹出时，若环境 **收缩布局视口**，则「视口底边」会整体上移，底栏会 **锚在可见区底边**（键盘上沿），表现为被「顶起」。

2. **`interactive-widget=overlays-content` 并非处处生效**  
   `index.html` 中已配置 `viewport-fit=cover` 与 `interactive-widget=overlays-content`（意图让键盘以 **覆盖** 方式出现、不整体推挤布局）。但在 **部分内置 WebView、部分 Android 浏览器** 中，仍可能出现 **布局视口随键盘变矮** 的行为。

3. **改 `100dvh` / `100lvh` / `100vh` 等无法从根本解决**  
   这类修改主要影响 **流式布局高度**；而底栏问题是 **`fixed` 相对「当前被压扁的布局视口」定位**。若不改变定位策略或做 **运行时补偿**，仅换高度单位 **通常无法消除**「底栏被顶起」的现象。

---

## 三、已验证有效的方案：运行时 + Visual Viewport

在 **登录成功、底部导航已挂载** 的 `MobileAppInner` 中，用 `useLayoutEffect` 对底栏节点（`ref`）做 **持续补偿**：

### 3.1 计算「应向下平移」的距离（inset）

- **优先使用 [Visual Viewport API](https://developer.mozilla.org/en-US/docs/Web/API/Visual_Viewport_API)**（在支持的浏览器中较可靠）：

  ```text
  insetBottom = window.innerHeight - visualViewport.height - visualViewport.offsetTop
  ```

  表示：布局视口底边相对 **当前可视区域** 的差值（键盘、地址栏变化等带来的 inset）。

- **回退（常见于「resize 型」键盘：整体 `innerHeight` 已变小）**  
  若上式得到的 inset **接近 0**，但键盘明显已打开，则使用：

  ```text
  insetBottom ≈ max(0, stableLayoutHeight - window.innerHeight)
  ```

  其中 **`stableLayoutHeight`** 在 **进入带底栏的壳层时** 取当时的 `window.innerHeight`，并在 **横竖屏切换** 后通过 `orientationchange` 延迟刷新，避免长期沿用错误基准。

### 3.2 应用到 DOM

对底栏设置：

```text
transform: translateY(insetBottom)   /* insetBottom > 阈值时 */
```

将底栏 **整体向下推**，使其重新贴在 **物理屏幕底缘**（可能被键盘遮挡；**不再**错误地「悬在键盘上方」）。卸载时 **清空** `transform`。

### 3.3 事件与时机

| 来源 | 作用 |
|------|------|
| `visualViewport` 的 `resize` / `scroll` | 随可视区域变化更新 inset |
| `window` 的 `resize` | 兼容部分环境 |
| `orientationchange`（延迟执行） | 刷新稳定高度并重新计算 |
| `document` 捕获阶段 `focusin`（`input` / `textarea` / `select`）后 **双 `requestAnimationFrame`** 再算一次 | 减轻键盘动画晚于焦点事件的抖动 |

## 四、曾尝试但未单独解决问题的方式（备忘）

- 将根节点 `min-height` 从 `100dvh` 改为 `100vh` + `100lvh` 等：在实测环境中 **未能** 消除底栏被顶起。
- 仅依赖 `interactive-widget=overlays-content`：在部分 WebView 中 **仍** 会resize 布局视口。

以上经验说明：**该问题要以「fixed 锚点 + 视口行为」为核心排查**，而不是只调全局高度单位。

---

## 五、本次新增经验（2026-03-30）

### 5.1 现象补充：看起来像“底栏又被抬起”

在弹幕输入场景中，曾出现「底栏三按钮再次可见并上浮」的回归现象，根因不止是键盘视口变化，还叠加了**层级覆盖关系**：

- `.mobile-bottom-nav` 的层级高于弹幕输入弹层时，输入弹层打开后底栏仍可见；
- 用户聚焦输入框后，视觉上会误判为“底栏被再次顶起”。

### 5.2 兜底策略（强建议保留）

在原有 `visualViewport + transform` 补偿基础上，增加两条与业务强相关的兜底：

1. **弹层层级高于底栏**  
   保证弹幕输入弹层始终盖住底栏（例如弹层 `z-index` 大于底栏）。
2. **弹层打开时临时隐藏底栏**  
   当 `danmakuOpen=true` 时，给底栏附加隐藏类（如 `opacity:0; pointer-events:none;`），关闭弹层后恢复。

这样可避免不同机型/浏览器的键盘行为差异继续影响输入体验，也能降低“已修复问题再次出现”的概率。

---

## 六、后续若仍有个别环境异常

若某浏览器 **既不缩小 `innerHeight`，`visualViewport` 也不反映键盘**，则上述公式可能失效，需针对该环境增加兜底，例如：

- 聚焦文本框时 **暂时隐藏底栏**（`visibility` / 类名切换），失焦后恢复；或  
- 该产品环境专用的 **UA / WebView 检测** 与单独策略。
