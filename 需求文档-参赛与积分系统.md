# 需求文档：参赛功能与选手积分系统

> 基于现有竞猜流程不变的前提下，新增「我要参赛」功能及选手积分体系。

---

## 一、需求概述

在手机端扫码打开的页面底部新增双入口：**我要竞猜** / **我要参赛**。竞猜流程保持不变，新增参赛报名流程；参赛选手采用积分制（**移除段位**），积分影响竞猜赔率。Unity 端及手机端竞猜页均展示双方选手积分与赔率。

---

## 二、功能需求详述

### 2.1 手机端底部导航

| 入口 | 说明 |
|------|------|
| **我要竞猜** | 跳转至现有竞猜页面，逻辑不变；展示双方选手积分与赔率 |
| **我要参赛** | 跳转至选手报名页面（见 2.2） |

---

### 2.2 「我要参赛」页面

#### 2.2.1 进入与可操作性

- **进入**：任何时候均可进入【我要参赛】页面
- **可操作条件**：仅当 `status === 'betting'`（竞猜已开启、报名进行中）时可操作
- **不可操作时**：展示对应状态文案，例如：
  - `waiting`：新一轮报名尚未开始
  - `started`：比赛正在进行中
  - `settled`：比赛已结束
- **历史战绩**：页面内新增【历史战绩】图标，点击可查看过往战绩（见 2.2.6）

#### 2.2.2 展示内容（可操作时）

- **玩法名称**：如「魔法冰球」「魔法拳王」
- **阵营选择**：用户可选择【红方】或【蓝方】
- **互斥规则**：红蓝双方各限 1 人；若已有其他用户选择某方，则该方不可再选

#### 2.2.3 操作按钮

| 按钮 | 行为 |
|------|------|
| **确认** | 将参赛数据同步至服务器，完成报名 |
| **撤销** | 撤销已确认的参赛数据，取消报名 |

**说明**：未点击「确认」前，参赛数据不落库、不同步。

#### 2.2.4 撤销时机

- 仅当 `status === 'betting'` 时可撤销
- `status === 'started'` 后不可撤销（与竞猜押注规则一致）

#### 2.2.5 互斥规则（参赛 ↔ 竞猜）

| 场景 | 拦截逻辑 |
|------|----------|
| 用户已有下注，点击「确认」参赛 | 拦截，提示：「当前有正在参与的竞猜，需要取消掉才能参赛」 |
| 用户已确认参赛，尝试参与竞猜 | 拦截，提示：「要先撤销参赛，才能参与竞猜」 |

#### 2.2.6 历史战绩

- **展示内容**：对阵双方、比分、积分变化、胜负结果、对阵时间
- **交互**：支持上下滑动（与历史竞猜记录类似）

---

### 2.3 选手积分体系

#### 2.3.1 基础规则

- 选手**无段位**，仅使用**积分**（段位体系已移除）
- 所有选手**初始积分 1500**
- 选手与竞猜用户**共用同一套账号体系**
- 积分用于：① 比赛结算时加减分；② 竞猜赔率计算

#### 2.3.2 比赛结算（加减分）

- **胜负**：胜方加分，负方减分
- **平局**：双方积分不变
- **变化量**：双方**比分差额**（如 3:1，差 2 分 → 胜方 +2，负方 -2）
- **零和**：胜方加分 = 负方减分

#### 2.3.3 竞猜赔率与积分

- **段位**：已完全移除，统一按积分计算赔率
- **赔率范围**：1.1 ~ 2.0，精确到小数点后 2 位
- **线性过渡**：积分差与赔率呈线性关系
- **规则对照**：

| 场景 | 红方赔率 | 蓝方赔率 |
|------|----------|----------|
| 双方积分相同 | 1.5 | 1.5 |
| 红方积分高 | 1.1 ~ 1.5 | 1.5 ~ 2.0 |
| 蓝方积分高 | 1.5 ~ 2.0 | 1.1 ~ 1.5 |
| 仅红方有选手 | 1.1 | - |
| 仅蓝方有选手 | - | 1.1 |
| 双方都无选手 | 1.1 | 1.1 |

---

### 2.4 展示信息（积分与赔率）

| 展示位置 | 内容 |
|----------|------|
| 手机端竞猜页 | 红方选手积分、蓝方选手积分、双方赔率 |
| Unity 端 | 红方选手积分、蓝方选手积分、双方赔率 |

---

## 三、数据与状态

### 3.1 参赛数据

- 红方选手：用户名（每局限 1 人）
- 蓝方选手：用户名（每局限 1 人）
- 报名状态：待确认 / 已确认 / 已撤销

### 3.2 选手积分

- 存储：用户名 → 积分（与 `users.json` 中的用户关联）
- 初始值：1500
- 更新时机：比赛结算时

### 3.3 历史战绩数据

- 对阵双方、比分、积分变化、胜负结果、对阵时间
- 需持久化存储，供历史战绩页展示

### 3.4 与现有竞猜的关系

- 竞猜下注时：校验用户是否已确认参赛，若已参赛则禁止下注
- 参赛确认时：校验用户是否有未结算的下注，若有则禁止确认

## 五、附录

### 5.1 状态与可操作性对照

| status | 参赛页可操作 | 竞猜可下注 | 可撤销参赛 |
|--------|-------------|-----------|-----------|
| waiting | 否 | 否 | 否 |
| betting | 是 | 是 | 是 |
| started | 否 | 否 | 否 |
| settled | 否 | 否 | 否 |

### 5.2 变更对照（原 → 新）

| 原概念 | 新概念 |
|--------|--------|
| 段位（红/蓝） | 移除 |
| 赔率矩阵（7×7） | 积分 → 赔率（1.1~2.0，线性） |
| 红/蓝段位展示 | 红/蓝选手积分 + 赔率展示 |

### 5.3 赔率公式（实现参考）

积分差与赔率线性映射，需在实现时确定**积分差阈值**（即积分差达到多少时赔率触及 1.1 / 2.0）。建议：

- 设 `d = |红方积分 - 蓝方积分|`，`dMax` 为阈值（如 400）
- 高积分方赔率：`1.5 - 0.4 × min(d / dMax, 1)` → 范围 [1.1, 1.5]
- 低积分方赔率：`1.5 + 0.5 × min(d / dMax, 1)` → 范围 [1.5, 2.0]
- 具体 `dMax` 值可在实现时调整
